# Humanize Architecture (V1)

Status: approved direction from product discussion on March 14, 2026.

## 1) Scope locked for MVP

- Microservices from day one.
- Event-driven communication using Aiven Kafka Free.
- Google login only.
- Upload books to cloud object storage.
- Support both text-based PDFs and scanned PDFs (OCR fallback).
- Notification channel: email only.
- Spoiler notification content generated by LLM from unread sections.
- Recommendations from user's private library only (public catalog later).
- Multi-device sync postponed, but schema/events prepared.

## 2) High-level system

1. React web client (desktop/mobile web first).
2. API Gateway (single public entrypoint).
3. Spring Boot microservices (private network):
- `auth-service`
- `library-service`
- `ingestion-service`
- `reader-service`
- `activity-service`
- `recommendation-service`
- `notification-service`
- `ai-service`
4. Kafka (Aiven Free) for async events.
5. PostgreSQL (single instance, schema-per-service).
6. Redis (cache + job dedupe + rate limits).
7. Cloudflare R2 (raw PDFs + extracted artifacts + covers).
8. Email provider (Brevo Free).
9. Gemini Developer API (free-tier model usage).

## 3) Service responsibilities

### `auth-service`
- Handles Google OAuth/OIDC callback and profile sync.
- Issues internal JWT access/refresh tokens.
- Publishes `user.authenticated`.

### `library-service`
- Creates upload intents (pre-signed URL to R2).
- Stores ownership, privacy, and processing state.
- Publishes `book.uploaded`.

### `ingestion-service`
- Consumes `book.uploaded`.
- Runs extract pipeline:
1. PDF text extraction (fast path).
2. OCR fallback (Tesseract) when text density is low.
3. Metadata normalization with `ai-service`.
- Writes normalized chapters/segments.
- Publishes `book.processing.completed` or `book.processing.failed`.

### `reader-service`
- Serves structured book content.
- Stores reading progress and checkpoints.
- Publishes `reader.progress.updated`.

### `activity-service`
- Captures reader events (`open`, `page_turn`, `session_end`).
- Builds daily engagement features for reminders/reco.
- Publishes `reader.activity.recorded`.

### `recommendation-service`
- Consumes progress/activity/book metadata (`BOOK_METADATA_EXTRACTED`).
- Produces ranked recommendations from private library only.
- Future: extend to public catalog when rights checks are ready.

### `notification-service`
- Stores user reminder windows and cadence.
- Generates reminder jobs from inactivity thresholds.
- Requests spoiler content from `ai-service`.
- Sends email through Brevo and publishes `notification.email.sent`.

### `ai-service`
- Central LLM adapter (single place for prompts, guardrails, fallback models).
- Generates:
1. Metadata cleanup (`title`, `author`, `genre` confidence).
2. Spoiler summary from unread segments with intensity controls.

## 4) Kafka topics (MVP set)

To fit Aiven Free limits, keep topic count small and stable:

1. `book.uploaded`
2. `book.processing`
3. `reader.progress`
4. `reader.activity`
5. `notification.lifecycle`
6. `user.lifecycle`

`book.processing` includes:
- `BOOK_PROCESSING_STARTED`
- `BOOK_METADATA_EXTRACTED`
- `BOOK_PROCESSING_COMPLETED`
- `BOOK_PROCESSING_FAILED`

Use an event `type` field inside each topic to avoid topic explosion.

## 5) Data model (schema-per-service, one Postgres instance)

- `auth.users`, `auth.sessions`
- `library.books`, `library.user_books`
- `ingestion.book_artifacts`, `ingestion.book_segments`
- `reader.progress`, `reader.bookmarks`
- `activity.events_daily`
- `recommendation.scores`
- `notification.preferences`, `notification.delivery_log`
- `ai.prompt_log` (masked/minimal)

## 6) Book processing pipeline

1. Client requests upload intent.
2. Client uploads PDF directly to R2.
3. `library-service` emits `book.uploaded`.
4. `ingestion-service` downloads and checks text density.
5. If density low, OCR path runs.
6. Extracted segments stored in R2 + Postgres index rows.
7. Metadata cleaned by `ai-service`.
8. Completion event updates library status to `READY`.

## 7) Spoiler-notification logic

1. Eligibility: inactive longer than user threshold and reminders enabled.
2. Select unread segments with highest "curiosity potential".
3. `ai-service` generates spoiler block (short, high-impact, personalized).
4. Email sent in user allowed time window.
5. Interaction events feed recommendation and reminder tuning.

## 8) Security and compliance baseline

- Google OIDC with backend token verification.
- Short-lived access JWT + refresh rotation.
- Encrypt sensitive values at rest (keys/SMTP/OAuth secrets).
- Signed URLs for upload/download; no public raw book URL.
- Copyright guardrail:
only user-owned/private books in MVP workflows.

## 9) Deployment (free-tier conscious)

- Aiven Kafka Free for event bus.
- Cloudflare R2 for object storage.
- Brevo Free for transactional/reminder email.
- Gemini Developer API free-tier models.
- App services on one low-cost VM or container platform initially.
- Postgres + Redis self-hosted on same environment for MVP cost control.

## 10) Known tradeoffs

- Microservices now increases operational complexity.
- Aiven Free Kafka limits throughput/topics, so keep event design compact.
- OCR is CPU-heavy; ingestion latency will vary by file quality.
- LLM-generated spoilers require prompt policy to avoid over-long excerpts.

## 11) Build order

1. Foundation: gateway, auth, library, R2 upload.
2. Ingestion: text extraction + OCR fallback.
3. Reader + progress + activity events.
4. Notification preferences + spoiler emails.
5. Recommendation service (private library only).
